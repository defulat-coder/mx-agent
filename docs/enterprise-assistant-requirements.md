## 四、跨域协作场景

利用 Router Team 的多 Agent 路由能力，实现跨部门流程的一站式服务。

### 4.1 新员工入职

```
员工: "我是新入职的，需要办什么手续？"

Router → 依次协调：
  HR Agent    → 入职手续清单、签合同、录指纹
  IT Agent    → 开通账号、领电脑、配权限、装软件
  行政 Agent  → 领工卡、分配工位、领办公用品
  财务 Agent  → 开通报销账户、绑定银行卡
```

### 4.2 员工离职

```
HR Agent    → 离职流程、交接清单、离职证明
IT Agent    → 回收设备、注销账号、回收权限
行政 Agent  → 退还工卡、清理工位
财务 Agent  → 结算报销、薪资结算
```

### 4.3 出差全流程

```
行政 Agent  → 差旅申请、订机票/酒店
HR Agent    → 考勤登记（出差状态）
财务 Agent  → 差旅报销、发票验真
```

### 4.4 项目启动

```
项目 Agent  → 创建项目、分配任务
HR Agent    → 调配人员、评估团队能力（人才发现）
IT Agent    → 开通项目相关系统权限
财务 Agent  → 设置项目预算
```

## 五、技术复用策略

所有新 Agent 复用现有基础设施，仅新增业务模型和工具：

| 基础设施 | 状态 | 复用方式 |
|----------|------|---------|
| JWT 认证 + 角色注入 | ✅ 已有 | 扩展 roles 枚举 |
| Router Team 路由 | ✅ 已有 | members 列表添加新 Agent |
| SQLite + AsyncSession | ✅ 已有 | 新增业务表 |
| Skills 知识库 | ✅ 已有 | 新增 skills/{domain}/ 目录 |
| 脱敏日志 | ✅ 已有 | 复用 masking.py |
| Tool 权限校验 | ✅ 已有 | 复用 get_xxx_id() 模式 |
| 会话记忆 | ✅ 已有 | AgentOS SqliteDb |
| 链路追踪 | ✅ 已有 | OpenTelemetry |

每个新 Agent 的开发模式统一为：

```
1. 定义数据模型 → app/models/{domain}/
2. 定义 Schema  → app/schemas/{domain}.py
3. 实现 Service → app/services/{domain}.py
4. 实现 Tools   → app/tools/{domain}/
5. 编写 Skills  → app/skills/{domain}/
6. 注册 Agent   → app/agents/{domain}_agent.py
7. 挂载到 Router → app/agents/router_agent.py members 追加
```

### 第二阶段：平台能力增强（📋 规划中）

以下功能围绕"从能用到好用"，按企业落地提效价值排序。

#### 1. 企业 IM 集成（P0 — 触达是提效前提）

当前仅有 REST API，用户需打开专门界面交互，触达率极低。

**目标**：接入企微/钉钉/飞书群聊和私聊，用户在日常 IM 中直接对话智能助手。

```
┌──────────┐   webhook    ┌──────────┐   /v1/chat  ┌────────┐
│ 企微/钉钉 │────────────▶│ 消息适配  │──────────▶│ Router │
│ 群聊/私聊 │◀────────────│ 层       │◀──────────│ Team   │
└──────────┘   回调       └──────────┘            └────────┘
```

**实现要点**：
- 新增 `app/adapters/` 目录，每个 IM 一个适配器
- 统一消息格式转换（IM 消息 ↔ Agent 消息）
- 群聊 @机器人 触发，私聊直接触发
- 长消息自动分段发送，支持 Markdown 卡片
- 核心 Agent 逻辑零改动

**数据模型**：无新增（复用 Agno 会话管理）

#### 2. 主动通知 & 定时提醒（P0 — 从被动查到主动推）

当前系统纯被动问答，合同到期、预算超支等关键事件无人主动通知。

**目标**：定时扫描业务数据，满足触发条件时主动推送通知。

```
┌──────────┐    定时扫描     ┌──────────┐    推送     ┌────────────┐
│ Scheduler │──────────────▶│ 触发规则  │──────────▶│ 通知渠道    │
│ (APSched) │               │ 引擎      │           │ IM/邮件/站内│
└──────────┘               └──────────┘           └────────────┘
```

**内置触发规则**：

| 规则 | 条件 | 通知对象 | 频率 |
|------|------|---------|------|
| 合同到期预警 | 到期前 30 天 | 法务人员 + 合同提交人 | 每日 |
| 预算超支预警 | 使用率 ≥ 80% | 部门主管 + 财务 | 每日 |
| IT 工单超时 | open 状态超 24h 未受理 | IT 管理员 | 每 4h |
| 员工合同续签 | 劳动合同到期前 60 天 | HR | 每周 |
| 会议室预订提醒 | 开始前 15 分钟 | 预订人 | 实时 |
| 快递到达提醒 | 新快递登记 | 收件员工 | 实时 |

**数据模型**：

| 模型 | 说明 |

| `NotificationRule` | 通知规则（名称/触发条件/通知渠道/接收角色/频率） |
| `NotificationLog` | 通知发送记录（规则ID/接收人/发送时间/状态） |

#### 3. RAG 知识库 — 企业文档检索（P1 — 扩展知识覆盖面）

当前 Skills 是手写 Markdown，内容固定且有限，无法覆盖企业几百页的制度文档。

**目标**：接入企业 PDF/Word 制度文档，通过向量检索增强 Agent 回答能力。

**实现方案**：
- 使用 Agno 原生 Knowledge Base（`PDFKnowledgeBase` / `DocxKnowledgeBase`）
- 向量库选型：PgVector（生产）或 LanceDB（轻量本地）
- 与现有 Skills 共存：Skills 做精确规则知识，RAG 做长尾文档覆盖
- 文档更新后自动重建索引

**适用场景**：
- 员工手册（几百页 PDF）
- 各类管理制度汇编
- 法规政策原文
- 产品/技术文档

#### 6. OA 审批流集成（P2 — 业务闭环）

当前请假/报销/差旅等操作只创建记录或返回 OA 链接，用户仍需跳转外部系统。

**目标**：打通 OA 审批系统，实现在对话中发起并跟踪审批。

**两种路径**：
- **A) Webhook 回调**：OA 审批完成后回写状态到 mx-agent
- **B) API 对接**：mx-agent 直接调用 OA API 发起/查询审批

**涉及审批流**：请假、加班、报销、差旅、用品申领、合同审批。

#### 7. 对话式报表 — Text2SQL Lite（P2 — 解决长尾查询）

当前每种查询都是写死的 Tool，管理者问"哪个部门加班最多"需要刚好有对应 Tool。

**目标**：针对现有 35 张表做受控自然语言查询。

**安全约束**：
- 白名单表和字段，禁止写操作
- 结果自动脱敏（复用 masking.py）
- 查询超时限制
- 仅限 admin/管理角色使用
